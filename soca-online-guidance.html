Executive Summary

SOCA, short for Security Operations and Camera Analytics, is an enterprise video analytics platform that uses YOLO-based inference on edge devices while keeping management, reporting, and multi-site visibility centralized.

The platform is built from four main services. At the edge, soca-engine handles RTSP ingestion, inference, rule evaluation, and alert publishing, while soca-dashboard provides the local web UI for cameras, schedules, rules, edge settings, and model management. In the server or cloud layer, soca-control acts as the fleet dashboard for alerts, reporting, edge visibility, and user management. soca-service is an optional consumer microservice used when the chosen transport pattern needs a separate bridge from Redis or Pub/Sub into the control plane.

Edge Processor soca-engine :8001
Edge UI soca-dashboard :8080
Control Plane soca-control :8000
Ingest Bridge soca-service :8010
Current implementation direction: the repository supports both on-premise and cloud-oriented deployments, with Redis Stream favored for local LAN setups and Google Pub/Sub plus GCS favored for multi-edge or managed cloud environments.

Key platform capabilities

  • YOLO model support for both PyTorch `.pt` and ONNX `.onnx` models.
  • Configurable inference performance through device selection, image size, and FP16.
  • Dedicated RTSP reader threading to decouple network I/O from inference.
  • Conditional LPR processing to avoid unnecessary OCR work when no relevant vehicles are present.
  • Alert category routing for intrusion, PPE, object detection, people counting, crowd, and LPR reports.
  • Live operator workflows through dashboard views, monitor pages, and centralized reporting.
  • Optional MediaMTX relay to avoid exposing raw camera credentials and to reduce direct camera connection load.

Service Roles

ServiceTechPortMain role
soca-engine FastAPI 8001 YOLO inference, job management, alert publishing, GCS snapshot sync.
soca-dashboard Django 8080 Per-edge UI for cameras, schedules, rules, settings, credentials, and local operations.
soca-service FastAPI 8010 Optional bridge that consumes transport messages and posts them into soca-control.
soca-control Django 8000 Fleet dashboard for all edges, alert persistence, reports, settings, and user management.

How the services interact

  1. soca-dashboard stores edge settings and can write them into the `soca-engine` environment file.
  2. soca-engine runs jobs against RTSP feeds, evaluates ROI logic and rules, and emits alerts.
  3. Those alerts move over Redis Stream or Google Pub/Sub.
  4. soca-control either consumes directly or receives alerts through soca-service.
  5. soca-control stores the resulting Alert records and exposes them through reports and dashboards.

Security model in the current docs

  • soca-control protects ingest with `SOCA_CONTROL_INGEST_KEY` for service-to-control API calls.
  • soca-engine is documented as running without authentication on a private network.
  • Cloud credentials for Pub/Sub and GCS can be uploaded through dashboard settings.
  • Licensing is handled through `soca-entitlement`, which issues UUID-based license keys and enforces camera and model limits.

Deployment Models

The docs support three practical deployment choices. The right choice depends on whether the target is a pilot, a managed enterprise host, or a cloud-first rollout.

Recommended default

Docker deployment

The Docker deployment guide is the fastest route to a repeatable environment. It also matches the repository split into `soca-edge`, `soca-control`, `soca-service`, and `soca-deploy`.

  • Good for demos, pilots, and standard server installs.
  • Central orchestration through `soca-deploy`.
  • Keeps edge and backend dependencies isolated.
Enterprise on-premise

Bare metal on RHEL or Rocky Linux

Use this path when hosts are managed under stricter enterprise controls or when the environment prefers direct service operations over containers.

  • Good for factory and facility infrastructure.
  • Fits traditional Linux administration patterns.
  • Pairs naturally with Redis Stream and local snapshots.
Cloud-first

GCP and Cloud Run

Use this path when centralized services belong in Google Cloud and the deployment already relies on managed infrastructure.

  • Good for distributed edges and managed services.
  • Usually paired with Pub/Sub and GCS.
  • Reduces local infrastructure for the control plane.

Docker deployment structure

~/projects/
  soca-edge/      # edge repo: soca-engine + soca-dashboard
  soca-control/   # fleet dashboard repo
  soca-service/   # ingestion bridge repo
  soca-deploy/    # orchestration repo

Core Docker prerequisites

  • Docker 24+ and Docker Compose v2.
  • Roughly 4 GB RAM available for the edge image because of YOLO and related dependencies.
  • Git access to all repos involved in the final topology.

Full Setup Flow

This section condenses the current `SETUP.md` into a single narrative. The platform is typically set up in this order: engine, dashboard, control plane, then optional service bridge.

Prerequisites

  • Python 3.11+
  • Git
  • Redis 7+ when using Redis transport
  • MediaMTX when using relay-based RTSP access
  • Google Cloud credentials when using Pub/Sub or GCS

Step 1: Set up soca-engine on the edge

git clone <repo> soca-engine
cd soca-engine
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt

Main engine environment settings:

EDGE_NAME=edge-jakarta-01
PUBLISHER_TYPE=redis
REDIS_URL=redis://localhost:6379
REDIS_STREAM_NAME=soca:detections

# Optional cloud transport
# PUBLISHER_TYPE=pubsub
# PUBSUB_PROJECT_ID=my-gcp-project
# PUBSUB_TOPIC=soca-detections
# PUBSUB_KEY_PATH=credentials/pubsub.json

# Optional snapshot storage
# GCS_BUCKET=your-bucket-name
# GCS_KEY_PATH=credentials/gcs.json
# GCS_PATH_PREFIX defaults to EDGE_NAME

# Performance
# INFER_DEVICE=auto
# INFER_IMGSZ=640
# INFER_HALF=false
# MAX_CONCURRENT_JOBS=4
./start.sh
# or
uvicorn main:app --host 0.0.0.0 --port 8001
curl http://localhost:8001/health
curl http://localhost:8001/models

Step 2: Set up soca-dashboard on the edge

git clone <repo> soca-dashboard
cd soca-dashboard
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt
python manage.py migrate
python manage.py createsuperuser
python manage.py runserver 0.0.0.0:8080

Main dashboard environment settings:

SECRET_KEY=your-secret-key-here
DEBUG=true
ENGINE_URL=http://localhost:8001
ENGINE_ENV_PATH=../soca-engine/.env
MEDIAMTX_URL=http://localhost:8888
MEDIAMTX_YML_PATH=../soca-engine/MediaMTX/mediamtx.yml
PORT=8080

Core dashboard actions after login:

  1. Open Settings and configure the edge connection, transport, and cloud storage details.
  2. Push the edge configuration so non-credential settings are written into `soca-engine` immediately.
  3. Push cloud credentials to the engine when using GCS or Pub/Sub.
  4. Add cameras with RTSP details.
  5. Create schedules with ROI configuration.
  6. Add rules with actions such as snapshot capture, queue publish, or Telegram notification.

Step 3: Set up soca-control on the server or cloud host

git clone <repo> soca-control
cd soca-control
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt
python manage.py migrate
python manage.py createsuperuser
./start.sh

Main control environment settings:

SECRET_KEY=your-secret-key-here
DEBUG=true
ALLOWED_HOSTS=localhost,127.0.0.1
# DATABASE_URL=postgresql://user:pass@localhost/soca_control
# SOCA_CONTROL_INGEST_KEY=<hex-string>
# GCS_BUCKET=soca-control-media-my-project
PORT=8000

Core control actions after login:

  1. Generate the ingest key in Settings if you are using soca-service as the bridge.
  2. Register each edge with its dashboard URL, engine URL, transport type, and GCS path prefix.
  3. Set the correct snapshot storage mode so alert images resolve correctly.
  4. Start the Redis consumer or Pub/Sub subscriber depending on the configured transport.

Step 4: Set up soca-service only when needed

In current docs, soca-service is the consumer bridge when alerts are not being consumed directly by soca-control. In cloud deployments using direct Pub/Sub subscription, soca-control can consume alerts itself and soca-service may be unnecessary.

Practical sequencing rule: do not start by documenting every optional integration. First get a single edge running with a camera, a schedule, a rule, and a visible alert in soca-control. Then add Pub/Sub, GCS, and reporting integrations.

Operations Guidance

The repository content implies three main user roles: deployment engineer, edge operator, and centralized control operator. The most effective way to use the platform is to keep each workflow narrow and explicit.

Deployment engineer

  • Installs services and runtime dependencies.
  • Chooses transport and snapshot storage model.
  • Configures environment variables and secrets.
  • Verifies health endpoints and end-to-end alert flow.

Edge operator

  • Adds cameras and confirms stream availability.
  • Creates schedules and ROI definitions.
  • Creates rules and local operational settings.
  • Starts, stops, and monitors jobs.

Central operator

  • Registers edges in soca-control.
  • Reviews alerts, snapshots, and reports.
  • Monitors fleet health across sites.
  • Manages users and branding.

MediaMTX relay model

MediaMTX is documented as the preferred RTSP relay strategy. Instead of giving every consumer a direct connection to the physical camera, the edge hosts a relay and consumers pull from `rtsp://localhost:8554/<camera_name>`.

  • Reduces the number of direct connections against the physical camera.
  • Keeps camera credentials on the edge instead of distributing them through job payloads.
  • Supports browser-facing HLS playback on port 8888.
  • Falls back to direct camera RTSP when relay is not configured.

Dashboard and Control Plane

soca-dashboard capabilities

The dashboard design documents show that the edge UI has grown beyond basic job start and stop into a broader operational console for cameras, rules, edge settings, snapshots, branding, models, and restart operations.

  • Authentication and role-based access model.
  • Camera management with stored RTSP credentials handled separately from the displayed URL.
  • Schedule management with Polygon, Rectangle, and Crossing Line ROI types.
  • Rule management for thresholds, event actions, and category-based reporting.
  • Settings for branding, edge configuration, user management, models, and operations.
  • Monitor page for live stream visibility and job state awareness.
  • Public edge info API and system resource display in the dashboard.

soca-control capabilities

soca-control is the fleet-level control plane. It stores alerts locally and provides reporting, filters, exports, and operator views across all registered edges.

  • Manual edge registration with dashboard URL, engine URL, transport type, and media relay settings.
  • Alert persistence into its own database for faster reporting queries.
  • Monitor page with direct links to edge media.
  • Report pages with statistics and detail tabs.
  • CSV and PDF export support.
  • User management and application branding.
  • Parallel polling of edge health and system endpoints to avoid UI hangs when sites are offline.

Control data model highlights

ModelPurpose
SiteConfig Singleton configuration for branding, snapshot storage type, and GCS bucket defaults.
Edge Registry of each site with URLs, transport settings, GCS prefix, media relay URL, and operational metadata.
Alert Normalized event record storing edge identity, camera details, rule info, snapshot path, ROI counts, and raw payload.

Engine API and Rules

The engine API guide provides repeatable examples for health checks, model listing, job startup, monitoring, ROI definitions, rule actions, and job teardown. These examples are the fastest way to validate the inference layer before relying on UI flows.

Common engine endpoints

EndpointPurpose
`GET /health`Service health, Redis status, active jobs, uptime.
`GET /models`Available model files.
`POST /jobs/start`Start an inference job.
`GET /jobs/{job_id}/monitor`Live annotated monitor stream.
`GET /jobs/`List all jobs.
`GET /jobs/{job_id}`Job status details.
`POST /jobs/{job_id}/stop` or stop action via UIStop a running job.

Minimal job start example

curl -X POST http://localhost:8001/jobs/start \
  -H "Content-Type: application/json" \
  -d '{
    "camera_id": "bardi-stairs-front",
    "rtsp_url": "rtsp://localhost:8554/Bardi-Stairs-Front",
    "model_path": "yolo/yolo11n.pt",
    "cls_ids": [0],
    "frame_interval_ms": 1000
  }'

Rule-capable example

curl -X POST http://localhost:8001/jobs/start \
  -H "Content-Type: application/json" \
  -d '{
    "job_id": "job-stairs-rules",
    "camera_id": "bardi-stairs-front",
    "rtsp_url": "rtsp://localhost:8554/Bardi-Stairs-Front",
    "model_path": "yolo/yolo11n.pt",
    "cls_ids": [0],
    "frame_interval_ms": 1000,
    "roi": {
      "type": "POLYGON",
      "points": [[100, 200], [600, 200], [600, 500], [100, 500]]
    },
    "rules": [
      {
        "name": "too_many_people",
        "priority": 10,
        "when_all": [
          {"path": "detections.in_roi_count", "op": "gte", "value": 3}
        ],
        "actions": [
          {"type": "save_snapshot"},
          {"type": "publish_queue"},
          {"type": "log", "level": "warning"}
        ]
      }
    ],
    "monitor": true
  }'

Detection modes and rule context

  • detection: fires when objects are detected in or out of ROI.
  • people_count: counts directional line crossings.
  • crowd: fires when the in-ROI person count exceeds a threshold.

Inference and performance notes

  • `.pt` models use `track()` and populate tracking identifiers.
  • `.onnx` models use `predict()` and do not provide tracking IDs in the same way.
  • Current documented device strategy is explicit CPU or CUDA selection through `INFER_DEVICE`.
  • `INFER_IMGSZ` and `INFER_HALF` are the first tuning knobs to use when optimizing latency.

Validation and Known Issues

The repository includes formal dashboard functional test results. The latest recorded dashboard test run shows a mostly working system with a number of UX and validation gaps that should be treated as release blockers for broader deployments.

StatusCount
PASS44
FAIL7
PARTIAL3
SKIP / N/A5
Total59

Most important documented bugs

  • Duplicate camera name can trigger an unhandled `IntegrityError` instead of a user-facing validation message.
  • Invalid or empty engine URL is accepted in edge settings without proper URL validation.
  • User creation can accept a blank password, which is a security issue.
  • A model that is still referenced by an active schedule can be deleted without warning.

Additional observations from testing

  • Some model files display `NaN KB` instead of actual file size.
  • Several flows surface raw HTTP errors to end users instead of clean UX messages.
  • Restart operations can time out from the browser perspective without first showing a clear “restart triggered” message.
  • Stopping an already-ended engine job can expose a raw `404` style message instead of a graceful status response.
Operational implication: this platform is already documentable and demoable, but the current test results indicate that form validation, destructive guardrails, and user-facing error handling still need another pass before relying on the dashboard as the only operator surface.
